## Project Goal

Build a search-first static knowledge base for `英雄技能 × 海克斯 / 攻击特效 / 法术效果 / 装备交互` questions in the new League of Legends `海克斯大乱斗` mode.

The current product direction is intentionally **not** “real-time AI answers”. It is:

- offline-curated structured data
- deterministic browser search
- short, high-confidence conclusions first
- explicit sources and confidence labels

## Product Decisions Already Locked

- Default interaction is direct search, not chatbot.
- Result cards should show short conclusion first, with only a little extra context.
- Data is AI-curated offline; user does not manually review entries.
- If one conclusion is clearly higher confidence, store and present that one.
- If sources conflict and neither side clearly wins, record both conclusions and mark conflict.
- Prefer `official / wiki / high-quality reference` sources.
- Third-party sites can be used as limited reference sources, but not mirrored or bulk-copied.

## Current Data Model

### 1. Manual entry

Used for high-value questions that need hand-shaped wording or stronger evidence.

Important fields:

- `id`
- `question`
- `aliases`
- `answer_short`
- `answer_detail`
- `status`
- `confidence`
- `patch_range`
- `conditions`
- `entities`
- `sources`

### 2. Derived entry

Generated from source tables. Current derived categories:

- champion skill `on-hit`
- champion skill `on-attack`
- champion skill `spell effects`
- champion skill `damage classification`
- champion skill `scaling note`
- augment directory entry
- augment rarity entry
- augment effect summary

## Search Behavior

Search is deterministic and intentionally biased toward quality:

- manual entries should rank above generated entries
- exact champion/skill/mechanic matches rank high
- alias normalization improves recall for names like `螃蟹 / 船长 / 狗头 / on-hit / 法术刃`

Important code:

- [scripts/lib/normalize.mjs](/Users/shenpeng.sp0916/go/src/personal/hex_brawl/scripts/lib/normalize.mjs)
- [site/search.js](/Users/shenpeng.sp0916/go/src/personal/hex_brawl/site/search.js)

Do not casually reintroduce broad substring bonuses; they previously caused generated entries to outrank better manual entries.

## What Is Already Working

- Static site can load and search generated data.
- Manual entries and generated entries are merged into one search index.
- Data validation runs before build.
- Derived entry generation is stable and tested.
- Homepage already has enough seed data for real use.

## Known Constraints

- This is still a structured knowledge site, not a full natural-language reasoning system.
- Most current generated champion interaction coverage comes from the public `attack effects` rules page plus curated summaries.
- Augment effect coverage is still partial: `25` summarized effects is useful, but far from full `195`.
- There is no automatic online fetch pipeline yet; source tables are still edited locally.

## Recommended Next Steps

Recommended order for the next AI:

1. Expand `data/source/augment-details.mjs` from `25` to `80-120` high-frequency augment summaries.
2. Expand `data/source/ability-mechanics.mjs` to cover many more champions and skill variants.
3. Add a dedicated `conflict` entry shape so the site can explicitly show “two conclusions + two source groups”.
4. Add more manual entries for the highest-frequency questions where generated text is too generic.
5. Add filter chips in the UI for `英雄 / 海克斯 / 机制 / 置信度 / 冲突`.
6. Add a lightweight offline “candidate extraction” script that helps propose new entries from raw notes, while keeping final storage structured.

## Good Immediate Work Items

If another AI takes over and wants low-risk, high-value tasks first:

- add more range augments / on-hit augments / rune augments / autocast augments to `augment-details`
- add more champions with mixed damage classification edge cases to `ability-mechanics`
- create `tests` for any newly important ranking rule before changing search ranking
- keep summaries short and query-oriented; users mostly want `能 / 不能 / 怎么判定`

## Style Rules For Future Entries

- Prefer short `answer_short` with a direct conclusion.
- `answer_detail` should explain the mechanism, not copy source text.
- Keep source links.
- Keep patch/version hints when available.
- If confidence is weak, do not silently phrase it as certain.
- If one source is much stronger, use that conclusion and mention the weaker disagreement only if useful.
